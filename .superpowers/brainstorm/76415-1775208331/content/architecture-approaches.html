Architecture Approach

How should we structure the MVP given we're deferring family settings, roles, and offline editing?

A

Full Schema, Minimal UI (Recommended)

Build the complete Supabase schema (families, members, lists, items) with RLS from day 1. But only build 3 screens: Login, Lists, List Detail. Family setup is a one-time guided flow after first login. No settings screen yet.

Pros

  • No schema migration later
  • RLS security from the start
  • Real-time works properly with family scoping

Cons

  • More upfront DB work
  • Family creation flow needed even for MVP
B

Flat Start, Migrate Later

Start with just lists and items tables. No family concept. All lists tied to a single user, shared via link. Add families/members when you need multi-user features.

Pros

  • Fastest to build
  • Simplest data model

Cons

  • Schema migration needed later
  • No real multi-user scoping
  • RLS harder to retrofit
C

Middle Ground — Family Implicit

Build families + items tables, but auto-create a family on first login. No explicit family management UI. Invite link adds members to the auto-created family. Expand UI later.

Pros

  • Correct schema without extra UI
  • Zero-friction onboarding
  • Invite link still works

Cons

  • Family name defaults to something generic
  • Slightly more backend logic for auto-creation
